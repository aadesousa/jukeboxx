Jukeboxx is a fork of Lidarr. The UI looks and works like Lidarr. The data
model, download pipeline, indexer system, and queue are modeled directly on
Lidarr's open-source architecture. The additions are:

  - Track-level monitoring (not just album-level)
  - Spotify as the primary source of truth for what you want
  - Soulseek as a first-class download client (alongside torrents and Usenet)
  - Spotizerr as a direct streaming source
  - YouTube as fallback
  - Playlist sync to Jellyfin (separate from download management)

GUIDING PRINCIPLES

1. BUILD ON LIDARR. Don't reinvent. Model downloads, indexers, quality profiles,
   and history exactly as Lidarr does them. Users who know Lidarr need zero
   learning curve.

2. VANILLA JS STAYS. No build toolchain. Single-file JS modules per page.

3. TEST AS YOU CODE. Every phase includes specific UI test steps that MUST be
   executed before marking complete. If a feature isn't tested, it isn't done.

4. PARITY FIRST. A working feature beats a polished stub. Don't add UI for
   something that has no backend, and don't add backend without wiring the UI.

LIDARR DOWNLOAD ARCHITECTURE — REFERENCE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Lidarr source: https://github.com/Lidarr/Lidarr
Key namespaces: NzbDrone.Core.Download, NzbDrone.Core.Indexers

DOWNLOAD CLIENTS (Lidarr model):
  Each client is a row in download_clients table:
    id, name, enable, priority, implementation, settings_json
  Implementations: QBittorrent, Sabnzbd, NzbGet, Deluge, Transmission,
                   RTorrent, Hadouken, etc.
  Each client has: host, port, username, password, category, recentPriority,
                   olderPriority, addPaused, useSsl, urlBase
  Test connection: ping the client, verify auth, verify category exists

  Jukeboxx adds: Spotizerr, MeTube, slskd (Soulseek), Prowlarr (as client)

INDEXERS (Lidarr model):
  Each indexer is a row in indexers table:
    id, name, enable, priority, protocol (torrent/usenet), implementation,
    settings_json, supportsRss, supportsSearch
  Implementations: Newznab (Usenet), Torznab (Torrent), Gazelle (music-specific)
  Prowlarr acts as a proxy — single Prowlarr entry gives access to all its
  configured indexers via standardized Torznab/Newznab API.

QUEUE (Lidarr model):
  Lidarr polls each enabled download client at regular intervals.
  Each item in the client queue is matched to a monitored album via:
    release title → fuzzy match against album name + artist name
  Queue items: id, title, status, protocol, size, sizeleft, eta, client_name,
               downloadId, album_id, artist_id, quality
  Actions: Remove, Blacklist, Mark as Complete

IMPORT (Lidarr model):
  On download completion, Lidarr:
  1. Finds the completed item in the download client
  2. Copies/hardlinks files to the music root
  3. Renames files using the naming format template
  4. Scans the new files for audio metadata
  5. Matches to monitored album → marks album status = 'have'
  6. Sends notification

HISTORY (Lidarr model):
  Every download event is logged: grabbed, downloadFailed, downloadFolderImported
  Per entry: date, album, artist, quality, indexer, downloadClient, protocol, data

3.1 Database schema changes

    Table: download_clients
      id INTEGER PRIMARY KEY AUTOINCREMENT
      name TEXT NOT NULL UNIQUE
      enable INTEGER DEFAULT 1
      priority INTEGER DEFAULT 1          -- lower = higher priority
      implementation TEXT NOT NULL        -- 'qbittorrent'|'sabnzbd'|'slskd'|
                                          --  'spotizerr'|'metube'
      protocol TEXT NOT NULL              -- 'torrent'|'usenet'|'soulseek'|
                                          --  'stream'|'youtube'
      host TEXT DEFAULT 'localhost'
      port INTEGER DEFAULT 8080
      use_ssl INTEGER DEFAULT 0
      url_base TEXT DEFAULT ''
      username TEXT DEFAULT ''
      password TEXT DEFAULT ''
      category TEXT DEFAULT 'music'       -- qBittorrent/SABnzbd category/tag
      recent_priority INTEGER DEFAULT 0  -- 0=last, 1=first (qBittorrent)
      older_priority INTEGER DEFAULT 0
      settings_json TEXT DEFAULT '{}'    -- implementation-specific extras
      created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
      updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP

    Table: download_history
      id INTEGER PRIMARY KEY AUTOINCREMENT
      date TIMESTAMP DEFAULT CURRENT_TIMESTAMP
      event_type TEXT                     -- 'grabbed'|'downloadFailed'|'imported'
                                          --  |'downloadIgnored'|'albumImported'
      monitored_album_id INTEGER
      monitored_track_id INTEGER
      source_title TEXT                   -- original release/filename
      quality TEXT                        -- 'flac'|'mp3-320'|etc
      indexer TEXT
      download_client_id INTEGER
      download_client_name TEXT
      download_id TEXT                    -- client-specific job/hash ID
      data TEXT                           -- JSON blob for extra fields
      created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP

    Table: indexers
      id INTEGER PRIMARY KEY AUTOINCREMENT
      name TEXT NOT NULL UNIQUE
      enable INTEGER DEFAULT 1
      priority INTEGER DEFAULT 1
      protocol TEXT NOT NULL              -- 'torrent'|'usenet'
      implementation TEXT NOT NULL        -- 'torznab'|'newznab'|'prowlarr'
      base_url TEXT NOT NULL
      api_key TEXT DEFAULT ''
      categories TEXT DEFAULT ''          -- comma-sep category IDs (3000=audio)
      supports_rss INTEGER DEFAULT 0
      supports_search INTEGER DEFAULT 1
      seed_ratio REAL DEFAULT 0           -- min seed ratio (torrents)
      seed_time INTEGER DEFAULT 0         -- min seed time in minutes
      created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP

3.3 Backend: download dispatch abstraction (downloader.py — new module)

    async def dispatch_to_client(client: dict, item: dict) -> dict:
      """Route a download item to the appropriate client implementation."""
      Routes based on client.implementation:
        'qbittorrent' → add magnet/torrent via qBittorrent WebUI API
        'sabnzbd'     → add NZB URL via SABnzbd API
        'slskd'       → POST /api/v0/searches + download best result
        'spotizerr'   → existing spotizerr_client.dispatch_download()
        'metube'      → existing youtube.py logic
      Returns: {ok, download_id, client_name, error?}

    async def get_queue_from_client(client: dict) -> list[dict]:
      """Fetch current queue from a download client."""
      qbittorrent: GET /api/v2/torrents/info?category=music
      sabnzbd:     GET /api?mode=queue&output=json&apikey=X
      slskd:       GET /api/v0/downloads (all active downloads)
      spotizerr:   GET /api/history/?limit=200
      Returns normalized list: [{id, title, status, size, sizeleft, eta, hash}]

    async def remove_from_client(client: dict, download_id: str) -> bool:
      Remove item from client queue.

    async def get_unified_queue() -> list[dict]:
      Iterates all enabled download clients, calls get_queue_from_client,
      merges results, annotates each with client_name and protocol.
      Returns sorted by priority (client priority first, then by added date).

4.1 Backend: dispatch endpoint

    POST /wanted/missing-albums/{id}/download
      1. Fetch monitored_album row
      2. Determine search terms: artist_name + album_name
      3. Search all enabled indexers (via indexers.py) → collect results
      4. Score each result vs expected album (title match + year + track count)
      5. Best result above threshold → dispatch to highest-priority enabled client
         If no indexer results (or no indexers) → try Spotizerr (if configured)
